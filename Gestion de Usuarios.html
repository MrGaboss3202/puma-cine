Gestión de usuarios

Flujo de autenticación y registro
Puma Cine
Proyecto · Bases de Datos
UNAM · 2026

Toda la lógica de cuentas de Puma Cine se apoya en una sola tabla del esquema WKSP_CINELISTA y en los endpoints AutoREST que Oracle APEX genera sobre ella. El registro y el inicio de sesión son dos operaciones distintas pero complementarias: la primera escribe en la base de datos respetando todas las restricciones de integridad, la segunda consulta esa misma información para validar al usuario en el cliente. Este documento describe cómo se articulan ambas y por qué cada decisión está atada a las restricciones declaradas en el esquema.

1La tabla USUARIO

El centro del sistema es la siguiente tabla:

CREATE TABLE USUARIO (
  ID_USUARIO       NUMBER         PRIMARY KEY,
  NOMBRE_USUARIO   VARCHAR2(...)  UNIQUE NOT NULL,
  NOMBRE_COMPLETO  VARCHAR2(...)  NOT NULL,
  EMAIL            VARCHAR2(...)  UNIQUE NOT NULL,
  CONTRASENA       VARCHAR2(...)  NOT NULL,
  ACTIVO           VARCHAR2(1)    CHECK (ACTIVO IN ('S','N')) NOT NULL,
  FECHA_REGISTRO   DATE           NOT NULL
)

Tres restricciones gobiernan todo el flujo:

  • Clave primaria sobre ID_USUARIO, generada por una secuencia + trigger BEFORE INSERT que asigna el siguiente valor con NEXTVAL.
  • UNIQUE sobre EMAIL y NOMBRE_USUARIO, que impiden cuentas duplicadas.
  • CHECK sobre ACTIVO, que sólo admite los literales 'S' o 'N'.

Cuando el cliente arranca, hace una llamada paralela a los ocho endpoints AutoREST que APEX expone para cada tabla del esquema. La respuesta de /usuario/ se guarda completa en memoria, incluida la columna CONTRASENA. Esto convierte a la tabla en el espejo local contra el que se valida cualquier inicio de sesión, sin necesidad de ir a la red en cada intento.

2Inicio de sesión

No existe un endpoint dedicado de login en la base de datos; esa lógica la resuelve el cliente con tres filtros encadenados sobre la copia local de la tabla, equivalentes a la consulta: 

SELECT *
FROM   USUARIO
WHERE  LOWER(EMAIL) = LOWER(:correo)
  AND  CONTRASENA   = :clave
  AND  ACTIVO       = 'S';

El formulario aplica las tres condiciones en orden y devuelve un mensaje distinto en cada falla:

  1. ¿Existe el correo? Si el filtro por EMAIL no encuentra ninguna fila, se muestra “No existe una cuenta con ese correo.” Esto cubre cualquier dirección que jamás pasó por un INSERT.
  2. ¿Está activa? Si la fila existe pero ACTIVO = 'N', se muestra “Esta cuenta está desactivada.” En el dataset actual, la usuaria paola_n está justo en ese estado y sirve como caso de prueba.
  3. ¿Coincide la contraseña? Si CONTRASENA no concuerda con lo escrito, se muestra “Contraseña incorrecta.” La columna almacena texto plano —en un sistema productivo se guardaría un hash— de modo que la comparación es directa, carácter por carácter.

Sólo cuando las tres condiciones se satisfacen se establece la sesión con la fila completa de USUARIO. A partir de ese momento, cualquier operación que necesite saber quién soy —insertar una reseña, agregar a Mi Lista, ver el perfil— usa el ID_USUARIO de esa fila como llave foránea hacia las tablas relacionadas.

3Registro de una cuenta nueva

Aquí sí hay una escritura real contra la base. El flujo del botón “Crear cuenta” se descompone en tres pasos:

A. Validaciones previas (lado cliente)

  • Los tres campos del formulario están llenos.
  • El correo no existe ya en la copia local de USUARIO: esta verificación anticipa la violación de la restricción UNIQUE para mostrar un mensaje amable en lugar de un código de error 400.
  • Se genera un NOMBRE_USUARIO único concatenando el primer nombre con un sufijo aleatorio de tres caracteres (por ejemplo demian_x7k), para no chocar con la otra restricción UNIQUE declarada sobre esa columna.

B. La petición POST a APEX

El envío equivale, en SQL, a la siguiente instrucción:

INSERT INTO USUARIO
  (NOMBRE_USUARIO, NOMBRE_COMPLETO, EMAIL, CONTRASENA, ACTIVO, FECHA_REGISTRO)
VALUES
  (:nombre_usuario, :nombre_completo, :email, :clave, 'S', SYSDATE);

ID_USUARIO no se envía: lo asigna el trigger BEFORE INSERT con NEXTVAL de la secuencia, garantizando una llave primaria irrepetible sin que el cliente tenga que conocer el siguiente valor disponible.

La parte interesante son los formatos exigidos por AutoREST, descubiertos a fuerza de errores reales durante el desarrollo:

ACTIVO debe ser exactamente la cadena "S". Enviar 1 o true rompe la restricción CHECK y Oracle responde con un 400 y el error ORA-02290: check constraint violated.
FECHA_REGISTRO debe venir en ISO 8601 con hora y zona horaria: "2026-05-22T19:34:56Z". Si se envía solamente la fecha "2026-05-22", APEX la parsea como nula y se viola el NOT NULL con ORA-01400: cannot insert NULL, exactamente el error que apareció en la primera prueba de registro.

C. La respuesta

APEX devuelve un JSON con la fila recién insertada, incluido el ID_USUARIO que generó la secuencia. El cliente toma ese identificador, construye el objeto de usuario en memoria y lo añade a la copia local de la tabla, de modo que la próxima vez que ese correo intente iniciar sesión ya esté disponible sin necesidad de recargar la página.

Si la inserción falla, el mensaje original de Oracle —restricción violada, columna NOT NULL, formato inválido— se propaga hasta la interfaz y se muestra debajo del botón. Esto permite distinguir a simple vista entre:

  • ORA-00001 — correo o nombre de usuario duplicado (UNIQUE).
  • ORA-02290 — valor inválido en ACTIVO (CHECK).
  • ORA-01400 — columna obligatoria sin valor (NOT NULL).

sin necesidad de abrir la consola del navegador.

4Por qué el diseño se sostiene en la BD

Cuatro decisiones de diseño hacen que el flujo sea robusto y que la base de datos —no el cliente— sea quien tiene la última palabra sobre la consistencia:

Integridad referencial garantizada por el motor

El cliente puede inventarse cualquier validación, pero quien decide al final es Oracle. Si un bug intentara insertar un correo duplicado, la restricción UNIQUE lo rechazaría incluso si el cliente no se diera cuenta.

Baja lógica vía la columna ACTIVO

En lugar de borrar usuarios físicamente —lo cual rompería las llaves foráneas de RESENA, USUARIO_PELICULA y USUARIO_AMIGO— se cambia ACTIVO a 'N'. La fila sigue existiendo para preservar el histórico de reseñas, pero la condición del login bloquea el acceso.

Clave primaria generada por secuencia

Permite que el cliente no tenga que conocer ni gestionar identificadores; basta con leer el que devuelve el INSERT. Esto evita condiciones de carrera y colisiones cuando dos usuarios se registran al mismo tiempo.

Separación de responsabilidades

La validación de credenciales correctas es lógica de negocio y vive en el cliente. La validación de datos íntegros vive en la base de datos a través de las restricciones declaradas en el esquema. Cada capa atrapa errores distintos y ninguna puede ser eludida por la otra.

En resumen: el registro es una inserción que atraviesa tres capas de validación —cliente → AutoREST → Oracle— y el inicio de sesión es una consulta con tres condiciones que se evalúan sobre la copia local de la tabla USUARIO. Todo el flujo está pensado para que ninguna columna quede inconsistente, ni siquiera ante peticiones malformadas o intentos maliciosos.
